# バーゲンセール 再発防止策 設計書（正本 v2・要承認）

> **v1→v2 の変更（SRE批判レビューを反映）**：最大の変更は **「受け入れ検査ゲートの単独出荷を禁止」**。
> ゲートだけを先に入れると、事故が `fresh-but-broken`（新しいが壊れてる）から
> `frozen-but-silent`（凍って誰も気づかない）に移り、**検知性はむしろ下がる**。
> これを塞ぐため、この設計が**自ら生む4つの新しい障害**を先に定義し（§3）、
> Phase 1 を「検知可能な安全網（detectable safety net）」の束として再定義した（§7）。
> あわせて「ゲート＝承認者」という呼称を撤回する。**承認者は人間の観測**であり、
> ゲートは沈黙を破る**サーキットブレーカ**にすぎない。

> **位置づけ**：正本。原則5の第1ゲート。**ユーザー承認後に実装へ**。対象は
> price 全滅事故（2026-07-06〜09）で露呈した「サイレント破損」の構造的再発防止。

## 1. 根本原因（1行）

> **制度は「入口＝人が書いた diff」を読むが、「出口＝毎日無人生成される本番データと、実際の画面」を誰も観測しない。**

- 壊れたのは bot が吐く `stocks.json`＝**誰の diff レビューにも回らない**。
- `update.yml` は中身を見ずに無条件 commit/push（検証ゼロ）。
- `except: return None`（全 get 関数）× アプリの null 非表示（`main.dart:615,621`）＝ **壊れても正常に見える**。
- `errors:[]` は各ヘルパーが例外を自分で飲むため、**中身がゴミでも真**。

---

## 2. 設計原則（改訂）

- **P1／出力に自動サーキットブレーカを置く**：無人パイプラインでは、閾値を割った出力の
  **昇格（＝配信）を機械的に止める**。ただし**ブレーカは"承認者"ではない**——沈黙を破って
  人間の観測を*呼び出す*装置。*Why：ブレーカを承認者と見なすとオートメーション・バイアス
  （「緑だから見なくていい」）が観測を殺す。今回、実際に効いたのは人間観測だった。*
- **P2／null を型で分ける**：「正常な欠損」と「取得失敗」を区別し、後者はアプリで**沈黙せず可視化**。
  *Why：原則4（不確実の可視化）を機械生成データにも適用。null はマーカー無しの究極の不確実。*
- **P3／"読む"より"動かして観測する"を上に置く**：`errors=0`・「例外が出ていない」を健全性の
  証拠にしない。**緑のゲートを人間観測の代替にしてはならない**。*Why：環境固有の真実
  （CIで `ticker.history` が空を返す）はテキストに存在せず、実行してしか観測できない。*

ゲートを素朴に入れると、事故は消えず**移動する**。移動先を先に塞ぐ。

## 5. L1 受け入れ検査 — 判定（3層）と閾値

**注意（過剰適合の自認）**：下の絶対フロアは現時点で**「正常1点＋全滅1点」の2スナップショット**にしか
校正できていない🟠。よって **絶対フロアは"全滅級"だけを確実に捕らえる緩めに置き、
微妙な劣化は前日比 `warn` で拾う**。数週間の観測分散が溜まったら引き直す（§8）。

**絶対フロア（割れたら `fail`＝配信停止）**：

| 検査 | 閾値 | 正常時 | 全滅 fb52f6a |
|---|---|---|---|
| price 充足率 | ≥ 90% | 99.7% | **0% → fail** ✅ |
| market_cap 充足率 | ≥ 90% | 99.7% | pass |
| 有効PER件数（`market_per` の母数）| ≥ 200 | 284 | 283 |
| `market_per` | not None | 18.62 | — |
| history 銘柄数 | ≥ ticker数×90% | 300 | — |
| universe_avg 点数 | ≥ 10 | 53 | — |
| スキーマ | 必須キー存在／`stocks` は配列／件数≒ticker数 | OK | — |

> 実データが示した要点：全滅時も PER は 283 件**生存**（`.info` は生きていた）。
> **PER件数ゲートだけでは見逃す→price 充足率ゲートが必須**。

**前日比 回帰（`warn` or `fail`）**：任意の主要フィールドの null 数が前日比 **+10ppt 超**で増加
（price null 2→300 = +99ppt）。ユニバース変更回（`tickers.json` の code 集合が変化）は**自動判定**で回帰を緩める。

**銘柄 quarantine**：フロアは満たすが個別に price 等が欠落する銘柄は、**スクリーニング対象から外し**、
`data_quality` に理由を残して配信（良品を巻き添え全凍結にしない）。

## 6. 実装アーキテクチャ

**`main()` を「生成 → 検証 → 合格時のみアトミック書込」に**：

1. stocks / histories / market を**メモリ上で**構築。
2. **既存コミット済み `stocks.json` を読み**、新データと比較（L2）。ベースラインが既知正常でない場合は seed から。
3. `validate(new, prev)` → 結果は `ok / warn / fail`。
4. **fail**：理由を print → **`sys.exit(1)`（ファイル上書きせず）** → `update.yml` の `if: failure()` が通知（N2）。前回の正本が生存。
5. **warn**：書込むが通知を出す。**ok**：静かに書込。
6. 書込は temp→`os.replace` で **3ファイル一括swap**（torn write 防止＝🟢7）。

**検証ロジックは `validate_stocks.py` に分離**し、**ゴールデンテスト（N3）を必ず同梱**。
**CIスモーク（L6a）** は別ジョブ：数銘柄を実 fetch → `price!=null` を assert（生成本体とは独立に環境バグを検知）。

## 7. 実装順序と段階ゲート（改訂）

**Phase 1 =「検知可能な安全網」（この束は分割出荷しない）**：
L1 ブレーカ（3層＋atomic）／L2 回帰（自動ユニバース）／**N1 鮮度バナー**／**N2 失敗通知**／
**N3 validatorゴールデンテスト**／**L6a CIスモーク**／N4 値サニティ／L5 依存固定。
→ これで「壊れても凍っても、**誰かに必ず届く**」状態になる。★最優先。

**Phase 2**：L4 `data_quality` 出力 → L3 アプリ欠損可視化 → **L6b 週1実機観測の義務化**。

**Phase 3（メタ）**：L7 CLAUDE.md 改訂。

**別トラック（配布判断が絡む・記録のみ）**：
- 🟢 **¥100 課金は廃止**（決定済・コンプラ）。ダミー「¥100」ボタン撤去、`kTestUnlockAll` はビルド構成で強制 false。
- ⏸ プロダクト是正（信号色廃止・「割安/割高」→中立表現・バリュートラップ警告・セクター相対・市場平均を幾何平均へ・免責の初回同意）＝**配布意図が決まってから**。
- 🟠 要専門家確認：金商法の投資助言業該当性、Yahoo/yfinance 規約（再配布・商用）。

各 Phase 完了時にユーザー承認を挟む（原則5）。
